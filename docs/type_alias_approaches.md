# Rosetta Type Alias: Python Generation Approaches

This document compares different strategies for handling Rosetta `typeAlias` definitions in the Python generator.

## Problem Statement
The Rosetta language supports `typeAlias`, which allows assigning a domain-relevant name to an existing type (e.g., `typeAlias ISIN: string`). Currently, the Python generator inconsistently ignores or replaces these aliases, leading to `NameError` exceptions in the generated code when an alias name is referenced but its definition hasn't been generated in the module.

---

## Comparison of Alternatives

### 1. Complete Replacement (The "Java Path")
*The generator expands every `typeAlias` to its underlying base type or Data class at generation time.*

| Pros | Cons |
| :--- | :--- |
| **Simpler Generator**: No need to track alias dependencies or manage cross-namespace imports specifically for aliases. | **Loss of Domain Meaning**: A field defined as `ISIN` in Rosetta becomes a plain `str` in Python. The developer's semantic intent is erased. |
| **Lower Runtime Complexity**: Zero overhead at runtime as there is no level of indirection. | **Opaque Schemas**: JSON schemas generated by Pydantic will show generic types (e.g., `string`) instead of domain types (e.g., `ISIN`). |
| **Robustness**: Reduces the chance of `NameError` or circular dependency issues specifically related to aliases. | **Divergence from Source**: The generated code feels less like the Rosetta source and more like an "optimized" or "compiled" output. |

### 2. Replication in Python (The "Rune Path")
*The generator creates actual Python assignments (e.g., `MyAlias = UnderlyingType`) and preserves the alias name in type hints.*

| Pros | Cons |
| :--- | :--- |
| **High Fidelity**: The generated code remains a faithful, one-to-one representation of the Rosetta model. | **Generator Complexity**: Requires a robust dependency graph (DAG) to ensure the underlying type is defined before its alias assignment. |
| **Better Developer Experience**: IDE tooltips and Pydantic-generated documentation will show domain-specific names (e.g., `test_alias_MyNumber`). | **Import Management**: Requires sophisticated cross-bundle import logic to ensure aliases are imported from their respective `_bundle.py` files. |
| **Future Proofing**: More in line with the Rune design philosophy of preserving model metadata in the target language. | **Namespace Pollution**: Adds more symbols to the top-level bundle modules. |

---

## Final Recommendation
While the "Java Path" is the current baseline in the Java generator, **the "Rune Path" (Replication) is the preferred choice for Python**. Python's native support for simple type assignments makes this a high-value, low-overhead win for code readability and developer productivity.

## Implementation Roadmap
To achieve the Replication approach, the generator will:
1.  **Generate Alias Assignments**: Scan models for `RosettaTypeAlias` and output the assignments in `_bundle.py`.
2.  **Preserve Hint Names**: Modify `PythonAttributeProcessor` to keep the alias name in field type hints.
3.  **Manage Dependencies**: Use a DAG to ensure the `UnderlyingType` is ordered before its `Alias`.
4.  **Handle Cross-Namespace Imports**: Ensure that when a function or type uses an alias from another namespace, the correct `import` statement is generated.
