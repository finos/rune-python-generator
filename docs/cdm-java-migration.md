## Analysis Summary

The Java-defined classes in the CDM library can be categorized by their interaction requirements:

1. **Platform/Environment Dependencies (OS = Y)**: Functions requiring access to the system clock (`Now`, `Today`) or core Java runtime internals (`hashCode` for key generation) are marked as OS-dependent.
2. **External Library Dependencies**: A significant number of scheduling functions (`CalculationPeriod`) and utilities (`AdjustableDateUtils`) rely on the **OpenGamma Strata** library. Replicating this in Rune is impractical due to the complexity of banking-grade holiday calendars and day-count logic.
3. **Infrastructure Orchestration & Utility Rationale**: Supporting classes (Group 3) are not recommended for Rune migration for three primary reasons:
    - **External Engine Dependencies**: Utilities like `AdjustableDateUtils` are wrappers for the **OpenGamma Strata** engine. Reimplementing these would require a complete banking-grade calendar and day-count engine in the DSL.
    - **Meta-Infrastructure Interaction**: These classes interact with low-level platform tools like the Rosetta `PostProcessor` and object pruning utilities. This is platform "plumbing" rather than domain-level business logic.
    - **Architectural Glue**: These classes manage the application lifecycle, including Dependency Injection (`@Inject`) and state orchestration across multiple workflow steps, which is outside the scope of the functional domain DSL.
4. **Ingestion Framework**: Ingestion processors handle the "dirty work" of parsing external formats. This logic is highly imperative, involves significant I/O (Jackson JSON mapping, Regex), and is better suited to Java's mature ecosystem for data transformation.

## Testing Proposal

### Testing Functions That Migrate to Rune
Functions migrated from Java to Rune (11 candidates identified) will be verified using **Rune-Native Post-Conditions**. This strategy leverages the DSL's declarative nature to embed expectations directly within the model.

#### Technical Implementation of Post-Condition Testing:
1.  **Embedded Assertions**: For each migrated function, `post-condition` blocks will be added to Rosetta files to define the expected relationship between inputs and outputs.
    *   *Example*: For `FpmlIrd8`, a post-condition would assert that if `party1 = party2`, then the count of distinct accounts in the output list must be exactly 2.
2.  **Generator Transformation**:
    *   **Java**: The Rosetta compiler generates `postProcess` methods or internal validation checks that throw `PostConditionException` if the logic is violated during runtime execution of the function.
    *   **Python**: The generator emits `assert` statements or Pydantic `@model_validator` calls that verify the result against the post-condition logic before returning the object.
3.  **Self-Verifying Integration Tests**:
    *   Instead of writing separate JUnit or PyTest logic for every business rule, we create "Test Runner" Rosetta functions. These functions take specific inputs (Golden Data) and call the target business function.
    *   The successful execution of the runner (no validation exceptions) confirms the function is behaving as expected on both platforms.
4.  **Benefits for Technical Review**:
    *   **Single Source of Truth**: The test logic is defined once in Rosetta and enforced in both Java and Python.
    *   **Auditable Logic**: Business analysts can review the `post-condition` in the model to ensure it matches the FpML or ISDA requirements without reading Java/Python code.
    *   **Runtime Safety**: These tests double as runtime guards, ensuring that even in production, the model cannot produce an invalid state that violates the defined business rules.

### Testing Native Programming Implementation
Functions that retain native implementations (22 functions requiring manual implementation in both Java and Python) follow a manual parity strategy:

1.  **Java Strategy**: The existing 22 implementations in Java will remain as the baseline. The primary change involves refactoring existing JUnit tests (approximately 22 test suites) to support data-driven execution using externalized test fixtures (JSON/YAML). This allows the same test cases to be reused by the Python implementation. The manually authored "Impl" classes in Java will continue to be the source of truth for the Java platform.
2.  **Python Strategy**: A corresponding Python implementation must be manually authored for each of the 22 functions. These implementations should be placed in the `rune.runtime` or a dedicated CDM-specific extension library.
3.  **Migration of Tests**: Approximately 22 sets of functional tests will need to be mirrored. The testing strategy relies on a "Test-Once, Verify-Both" approach where test fixtures (input parameters and expected results) are defined in a language-agnostic format (like YAML or JSON). Both the Java JUnit suite and the Python `pytest` suite will ingest these same fixtures to ensure that the manually written Python logic exactly matches the mature Java logic.
